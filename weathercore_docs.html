Phase 1 — Database Foundation

A normalised MySQL 8.0 schema with six tables stores every weather observation, station metadata, pre-aggregated daily summaries, alert events, API call logs, and runtime configuration. Composite indexes on (station_id, observed_at) make time-series queries fast even at millions of rows.

Phase 2 — Automatic Update Pipeline

A Python background process fetches current conditions from Open-Meteo every 15 minutes for all active stations. Each response is normalised, deduplicated on a unique key, stored in the observations table, and passed to the alert engine. A nightly cron-style job aggregates raw hourly observations into the daily_summaries table via a MySQL stored procedure.

Phase 3 — Interface Integration

A Flask REST API exposes the database over HTTP. The dashboard — a single self-contained HTML file — consumes these endpoints. When Flask is unreachable it falls back to the Open-Meteo API directly from the browser, so the UI always renders with data.

sudo apt update && sudo apt install mysql-server -y

Section 04

Schema Design

Six tables, two views, and one stored procedure. The schema is fully normalised — station metadata is stored once and referenced by foreign key in every fact table.

Table: stations

Master reference for every monitoring location. Every observation, alert, and daily summary references this table via station_id.

Indexes Two indexes are critical for performance: idx_station_time (station_id, observed_at) for station-specific time-range queries and idx_time (observed_at) for system-wide temporal queries. Both are created by the schema file.

Table: daily_summaries

Pre-aggregated daily statistics computed nightly from raw observations by the stored procedure sp_compute_daily_summary. This table exists to serve fast chart queries — rather than running GROUP BY over millions of hourly rows at request time, the frontend reads from pre-computed rows in this table.

Table: alerts

Every threshold breach creates a row. An alert is considered active when resolved_at IS NULL. The system prevents duplicate active alerts of the same type per station using a has-active-alert check before insertion.

alertsThreshold breach event log
ColumnTypeDescription
alert_typeENUMheat | cold | storm | flood | wind | uv | pressure | visibility
severityENUMwatch | warning | emergency
metric_nameVARCHAR(40)Field that breached the threshold
metric_valueDECIMAL(10,3)Actual sensor value at time of breach
threshold_valueDECIMAL(10,3)Configured threshold that was crossed
triggered_atDATETIMEUTC timestamp of the first breach
resolved_atDATETIME NULLNULL = still active; set by POST /alerts/:id/resolve

Table: api_log

An audit trail for every outbound HTTP call made by the ingestion engine. Tracks latency, HTTP status, and error messages per source. Used by the GET /api/system/stats endpoint to return API health metrics to the dashboard.

Views

Two views simplify the most common read patterns:

ViewReturnsUsed by
v_latest_conditionsMost recent observation per active station, joined with station metadataGET /api/current
v_active_alertsAll unresolved alerts ordered by severity then time, joined with station namesGET /api/alerts/active

Stored Procedure: sp_compute_daily_summary

Called by the nightly summariser with (station_id, date). Computes min, max, mean temperature, total precipitation, peak wind, and observation count for the target date using a single aggregate query, then upserts the result into daily_summaries using INSERT ... ON DUPLICATE KEY UPDATE.

Deduplication

The INSERT IGNORE statement with the unique key (station_id, observed_at, source_api) ensures that re-running an ingestion cycle for the same time window never creates duplicate rows. This makes the scheduler safe to restart without data corruption.

Error Handling

Each station's fetch is wrapped in an independent try/except block. A failure for one station is logged to api_log and the cycle continues for the remaining stations. The scheduler never stops due to a single station failing.

⚠
Flask Reloader Always start Flask with use_reloader=False in production and development. The Werkzeug reloader runs the application twice, which causes APScheduler to register duplicate jobs and double all API calls.
Section 08

Alert Engine

Threshold rules are stored in system_config and reloaded hourly. Every successful observation triggers a check against all five rule types.

Alert TypeField CheckedWatch / Warning / Emergency (defaults)
heattemperature_c >— / 35°C / 42°C
coldtemperature_c <— / 0°C / −10°C
windwind_speed_ms >— / 15 m/s / 25 m/s
flood / rainprecipitation_mm >— / 10 mm/h / 30 mm/h
uvuv_index >— / 7 / 10

Thresholds are configurable at runtime from the dashboard's Alerts page — no code change or server restart required. The alert engine runs on both the backend (inserting to the alerts table) and on the frontend (client-side evaluation for immediate visual feedback when Flask is offline).

Section 09

Flask REST API

All responses are JSON. decimal.Decimal, datetime, date, and timedelta types returned by MySQL are automatically serialised by the custom _serial() handler.

✦
CORS Flask-CORS is configured to allow all origins on /api/* routes, permitting the dashboard HTML file to be opened directly from the filesystem (file://) without needing to serve it from the same origin.

Section 11

Dashboard Interface

A single self-contained HTML file. No build step, no npm, no framework. Opens directly in a browser. Connects to Flask when available; falls back to Open-Meteo when offline.

🏙️
City Management
Search and add any city worldwide via geocoding. Cities persist in the sidebar with temperature readouts. Remove or toggle compare per city.
⊕
Multi-city Compare
Click ⊕ on any city to add it to the comparison view. Sortable table, colour-coded bar charts, hot/cold row highlights.
📈
Historical Charts
Four separate charts: temperature, precipitation, wind, humidity. Dynamic ranges: 6h, 24h, 3d, 7d, 30d. Each chart expands to full-screen modal.
🔔
Alerts Page
Live alert list from the database. Configurable thresholds for heat, cold, wind, rain and UV — updated without server restart.
🗺️
Station Map
Leaflet dark map with temperature markers. Marker colour reflects alert level. Click any marker to switch active city.
⤢
Chart Expand
Every chart has an expand button. The modal includes its own range selector so you can change time window without closing it.

Overview page

The default landing view. Shows current conditions for the selected city in a hero card with animated KPI strip (humidity, wind, pressure, UV, cloud cover, visibility), a temperature trend chart, and a precipitation/humidity dual-axis chart. The station map occupies the bottom-right quadrant.

Compare page

Any city marked with ⊕ in the sidebar appears here. Shows per-city condition cards with WMO weather icons, a sortable table comparing all key metrics, and two bar charts for temperature and humidity. Red row border indicates the hottest city; blue indicates the coldest.

Historical page

Data for the currently selected city. A stat row at the top shows min/max/avg temperature and total precipitation for the selected period. Below are four expandable charts. The range selector (6h through 30d) refetches data from Flask or Open-Meteo and re-renders all four charts simultaneously.

Alerts page

Full-width alert cards with severity colour-coding (red border = emergency, amber = warning, blue = watch). Below the live alerts is a threshold configuration panel — 10 input fields covering all five alert types at both warning and emergency levels. Clicking Save re-runs the alert engine against all current city data immediately.

Offline / Demo mode

On boot the dashboard probes Flask with a 2-second timeout. If unreachable, it loads six pre-configured East African cities with demo weather data and switches to fetching directly from Open-Meteo. A source badge in the topbar shows either ⬡ MySQL or ⬡ Open-Meteo to indicate the active data source.
