Official Project Documentation

Global Weather
Database Consortium

A collaborative, open-stack initiative to build persistent, queryable, multi-station weather monitoring infrastructure — making high-quality atmospheric data accessible, affordable, and actionable at every scale of decision-making.

Full Name Global Weather Database Consortium
Abbreviation GWDC
Document Version 1.0 — Initial Release
Stack Python · Flask · MySQL · Open-Meteo
Primary Data Source Open-Meteo (free, no registration)
Classification Open Source · Free Stack
The GWDC project demonstrates that a production-grade, multi-city weather monitoring system — with persistent storage, automated ingestion, real-time alerting, and an interactive analytics dashboard — can be built entirely from free, open-source tools by a small team. It exists to close the gap between expensive commercial environmental data platforms and the institutions — governments, universities, NGOs, farmers — who need this data most but can afford it least.
Section 01

Mission & Objectives

The GWDC was established around a single, concrete problem: high-quality, persistent weather data is expensive to acquire and technically complex to manage, creating a systematic disadvantage for low-resource institutions in data-dependent decision-making.

The problem being solved

Commercial weather data platforms — PlanetOS, Tomorrow.io, DTN — cost thousands of dollars per year per location and require technical integration work that exceeds the capacity of most public-sector organisations in developing regions. Free weather APIs like Open-Meteo solve the cost problem but not the persistence problem: they return current conditions and short forecasts, but provide no mechanism for storing readings over time, querying historical trends, or receiving automated alerts when dangerous thresholds are crossed.

The GWDC project solves both problems simultaneously — by building the infrastructure layer that sits between a free API and a useful, long-running data asset.

Section 04

Phase 1 — Database Foundation

The database is the most critical component of the system. Everything else — the pipeline, the API, the dashboard — is infrastructure built to write into and read from it.

Schema design philosophy

The schema follows third normal form (3NF). Station metadata is stored once in the stations table and referenced by foreign key everywhere else, ensuring that correcting a station's coordinates or name requires a single row update, not a full-table rewrite. The observations table is the primary fact table — optimised for high-volume writes and fast time-range reads via composite indexing on (station_id, observed_at).

Data ingestion strategy

Each ingestion cycle queries the stations table for all rows where active = 1, then fetches current conditions from Open-Meteo for each station independently. The response is normalised — field names mapped from Open-Meteo conventions to database column names, units converted where necessary — and inserted using INSERT IGNORE against a unique key on (station_id, observed_at, source_api). This deduplication strategy makes the scheduler safe to restart, re-run, or overlap without producing duplicate data.

The raw_json column in observations stores the full, unmodified API response for every row. This provides a complete audit trail — if a normalisation bug is discovered later, the raw data can be reprocessed without re-fetching from the API.

Section 05

Phase 2 — Automatic Update Pipeline

Scheduler

APScheduler runs three background jobs within the same Python process as the Flask server, eliminating the need for a separate cron daemon or task queue for basic operation. The ingestion job fires every 15 minutes and processes all active stations in sequence. The nightly summary job runs at 00:05 UTC and calls the stored procedure sp_compute_daily_summary for each station, populating daily_summaries from the previous day's raw observations. A third job reloads alert thresholds from system_config every 60 minutes, ensuring that changes made via the dashboard take effect without a server restart.

Alert engine

After every successful observation insert, the alert engine evaluates five independent threshold rules — heat, cold, wind, flood/rain, and UV. Each rule has two severity levels: warning and emergency. Before inserting a new alert, the engine checks whether an unresolved alert of the same type already exists for that station, preventing duplicate notifications from stacking. Alerts are stored in the alerts table with a resolved_at timestamp of NULL until manually resolved via the dashboard or the REST API.

⚠
Threshold defaults are conservative Default thresholds (heat warning at 35°C, wind emergency at 25 m/s) are set for general tropical conditions. Deployments in polar, alpine, or desert environments should reconfigure thresholds via the Alerts page before going live.
Section 06

Phase 3 — Interface Integration

REST API

Flask serves twenty endpoints across four categories: core data endpoints (stations, current conditions, history, summaries), dashboard bridge endpoints that translate database column names to Open-Meteo field name conventions, alert management endpoints, and system administration endpoints. All responses are JSON. A custom serialiser handles MySQL-specific types — decimal.Decimal to float, datetime to ISO 8601 string — ensuring consistent, browser-parseable output from every endpoint.

CORS is enabled on all /api/* routes, permitting the dashboard HTML file to be opened directly from the filesystem during development without requiring a same-origin setup.

Dashboard

The dashboard is a single self-contained HTML file requiring no build step, no package manager, and no framework. It loads Chart.js and Leaflet from CDN links. On boot it probes Flask with a two-second timeout — if Flask responds, all data flows from MySQL. If Flask is unreachable, the dashboard loads six default cities with demo data and switches to fetching directly from Open-Meteo. A source badge in the topbar indicates which mode is active.

The dashboard provides six pages: Overview (current conditions hero card, KPI strip, and mini charts), Compare (multi-city sortable table and bar charts), Historical (four expandable time-series charts with 6h/24h/3d/7d/30d range selector), Alerts (live alert list and configurable threshold panel), and two stub pages (Analytics and ML Predictions) reserved for future development.

Section 07

Usefulness of the System

The GWDC system's value is not simply that it displays weather data — browsers can do that. Its value is that it remembers weather data, analyses it, and acts on it automatically, indefinitely, without human supervision.

Why persistent storage changes the value equation

Every free weather API returns only the current moment or a short forecast window. The instant the HTTP response closes, that observation is gone. GWDC interrupts this pattern by writing every reading to MySQL — creating a locally owned, queryable archive that accumulates value with every passing week. After 90 days at eight stations, the system holds approximately 207,000 individual observations across 18 variables. After one year it holds roughly 830,000. This archive enables a category of analysis that no API call can provide: trend detection, seasonality analysis, anomaly identification, and eventually machine learning.

"A weather API tells you what the atmosphere is doing right now. GWDC tells you what it has been doing for the past year, across multiple cities, and flags when anything dangerous is developing — without anyone having to watch."

The alert engine as an operational multiplier

Without an alert layer, a monitoring system requires a human to watch it. With one, a single deployment can monitor eight or eighty stations continuously and notify decision-makers only when action is required. This transforms GWDC from a data display tool into operational infrastructure — the difference between a thermometer and an alarm system.

Alert thresholds are configurable at runtime from the dashboard without touching code or restarting the server, making the system adaptable to different climates, operational contexts, and risk tolerances without requiring developer intervention.

Multi-city comparison as a spatial analytics tool

The comparison page allows simultaneous monitoring of multiple locations with a sortable table that highlights the hottest, coldest, and most humid city at any moment. For organisations operating across a geography — regional health authorities, agricultural cooperatives, energy operators, logistics companies — this view replaces the manual process of checking weather for each location individually with a single, ranked, colour-coded summary.

Government & Disaster Risk Management

National and county governments bear primary responsibility for disaster early warning. GWDC provides the monitoring infrastructure that feeds early warning systems at a fraction of the cost of commercial platforms — critical in budget-constrained public-sector contexts.

  • Flood precursor detection — precipitation rate exceeding 30 mm/hour triggers evacuation preparation in low-lying areas

Section 09

Technical Challenges

Building a production-grade data system from free tools surfaces a predictable set of engineering difficulties. Understanding these challenges — and how they were resolved — is as instructive as the system itself.

01
MySQL Type Serialisation
MySQL's Python connector returns DECIMAL columns — including latitude, longitude, temperature, and all numeric measurements — as Python decimal.Decimal objects rather than floats. The standard json.dumps() function cannot serialise these, causing every Flask endpoint to raise a TypeError on any response containing numeric data. The fix required a custom _serial() function registered as the default JSON encoder, handling Decimal → float, datetime → ISO string, timedelta → seconds, and bytes → hex — covering all MySQL column types that are not natively JSON-serialisable.
Technical
02
Boot Deadlock — Loading Screen That Never Clears
The initial dashboard boot sequence fetched weather data for all six default cities sequentially using await, meaning each city's HTTP request had to complete before the next began. If any single Open-Meteo request hung — which occurs under rate limiting or poor connectivity — the entire boot function stalled indefinitely. The loading overlay never dismissed because it was gated on the completion of network requests. The resolution involved three changes: a hard 1.5-second setTimeout that dismisses the loading screen unconditionally; wrapping all fetch calls in an AbortController with per-source timeouts (4s for Flask, 7s for Open-Meteo); and switching from sequential await to parallel Promise.allSettled() so a single station failure cannot block the others.
Technical
03
iframe Cross-Origin Blocking
The documentation was designed to load inside an iframe within the dashboard. When both files are opened directly from the filesystem using the file:// protocol, browsers enforce a same-origin policy that silently blocks the iframe from rendering its target. The iframe element receives the src attribute and the network request is made, but the browser discards the response — resulting in a blank frame with no error visible to the user. The resolution is to serve both files through Flask (http://localhost:5000), placing them on the same HTTP origin and eliminating the restriction. This is why the documentation tab functions correctly when Flask is running but appears blank when files are opened directly in the browser.
Technical
04
APScheduler and Flask Reloader Conflict
Flask's development server includes a Werkzeug auto-reloader that, by default, runs the application process twice — once as a monitor process and once as the actual worker. When APScheduler starts in the application module, it therefore starts twice, registering every job twice and doubling all API calls, database writes, and alert evaluations. This produces duplicate observations for every station on every ingestion cycle. The fix is a single line in the Flask startup call: use_reloader=False. In production this is not an issue as Gunicorn and similar WSGI servers do not use a reloader.
Operational
05
Chart.js Time Scale with Mixed Data Sources
The historical charts use Chart.js with a time type x-axis, which requires the chartjs-adapter-date-fns CDN script to be loaded separately from Chart.js itself. When data comes from Flask, timestamps are ISO 8601 strings (e.g. 2025-07-01T14:00:00). When data comes from Open-Meteo directly, timestamps are naive strings without timezone designators (e.g. 2025-07-01T14:00). Constructing JavaScript Date objects from naive strings produces inconsistent timezone interpretation across browsers. The normalisation layer appends Z to naive strings before constructing Date objects, ensuring UTC interpretation in all browsers.

Section 10

Data Challenges

06
API Rate Limits and Gap Prevention
Open-Meteo permits 10,000 requests per day per IP address. At 8 stations, one fetch per station per ingestion cycle, and 96 cycles per day, the system makes 768 requests per day — well within the limit. However, adding a secondary source (OpenWeatherMap at 1,000 requests/day) constrains the architecture: the secondary source can only be polled for a subset of stations or at a lower frequency. The api_log table was designed specifically to track this — the /api/system/stats endpoint exposes per-source call counts and success rates over the last 24 hours, enabling proactive detection of approaching rate limits before gaps appear in the time-series.
Data
07
Field Name Convention Mismatch Between Sources
Open-Meteo uses snake_case field names like temperature_2m and relative_humidity_2m. The MySQL schema uses more concise names like temperature_c and humidity_pct that include the unit suffix. The Flask dashboard bridge endpoints were built specifically to resolve this mismatch — translating DB column names to Open-Meteo conventions so the frontend JavaScript functions identically regardless of data source. Without this translation layer, switching the dashboard between Flask and direct Open-Meteo mode would require different rendering code for each source, doubling the frontend complexity.
Data
08
PyCharm SQL Data Source Configuration
PyCharm's SQL editor displays a warning on .sql files — "No data sources are configured to run this SQL and provide advanced code assistance" — when the IDE has no knowledge of the target database. This is a development environment configuration issue, not a code error. Resolution requires opening the Database panel in PyCharm (View → Tool Windows → Database), adding a MySQL data source with the project's connection details (localhost, port 3306, weatheruser, weathercore), downloading the JDBC driver when prompted, and clicking Test Connection. Once configured, PyCharm indexes the schema and provides full SQL autocomplete, inline error detection, and the ability to run queries directly from the editor.

Section 11

Operational Challenges

09
Keeping the System Running Without Commercial Infrastructure
A production deployment requires the Flask process to stay running indefinitely — surviving server reboots, process crashes, and memory pressure. On a local machine this is straightforward, but on a bare server or cloud VM it requires a process manager. On Linux, systemd can be configured to run python app.py as a managed service that restarts on failure and starts at boot. On Windows, the NSSM (Non-Sucking Service Manager) utility provides equivalent functionality. Without a process manager, the system stops collecting data every time the terminal session closes or the machine reboots, creating gaps in the time-series that cannot be backfilled.
Operational
10
Database Growth and Long-term Storage Planning
The observations table grows at approximately 2,300 rows per day at 8 stations with a 15-minute ingestion interval. After one year this is roughly 830,000 rows; after five years, 4.2 million. Without table partitioning and a retention policy, query performance will degrade progressively. The system_config table includes retention_days_raw and retention_days_summary keys, but enforcement — a scheduled DELETE statement removing observations older than the configured window — is not yet implemented as a MySQL Event. This is a known gap requiring a follow-on development task before the system is deployed for multi-year operation.

Section 14

Impact Statement

◈
Core finding A production-grade, multi-city weather monitoring system with persistent storage, automated ingestion, real-time alerting, and an interactive multi-page dashboard can be built entirely from free, open-source tools by a small team in a matter of weeks. The GWDC project proves this claim with a working system.

The significance of that finding extends beyond weather monitoring. It demonstrates that the infrastructure gap between well-funded institutions and under-resourced ones is, in the domain of environmental data, closeable with engineering skill alone — not capital. The same free APIs, the same open-source database, the same Python libraries are available to a county government in a rural area of Kenya as to a multinational company in London. What differs is the technical capacity to build the infrastructure layer that makes them useful.

The GWDC project builds that capacity. In doing so it produces, as a by-product of its own construction, the very skills that would be needed to adapt it to any other domain — agriculture, health, energy, transport, finance — where the same monitoring pattern applies and where the same infrastructure gap exists.
